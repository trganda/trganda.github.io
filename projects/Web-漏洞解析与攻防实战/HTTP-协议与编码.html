        HTTP 协议与编码

        Jul 28, 20243 min read

        GET 参数

        • 一般形式
          • key = value
        • 数组形式
          • key[name] = value

        编码

        请求体中传输的内容，有多种形式，如传输文件一般使用 multipart/form-data，或数据量较大时『如：视频传输』采用 chunked 形式进行传输。

        chunked

        常见请求报文形式

        通过 Transfer-Encoding 指定 chunked 模式，请求体中 1a 表示长度声明，为十进制的 26。其后下一行的内容 a=To be or not to be, that 为传输的数据。后面的内容依旧是 长度 + 内容 的形式。

        当长度为 0 表示传输结束，其后跟两个 \r\n。

        multipart

        通过 Content-Type 指定请求头格式，字段值的形式为

        Content-Type: multipart/form-data; boundary=----WebKitFormBoundary{随机字符串}

        之后的内容以 ------WebKitFormBoundary{随机字符串} 作为开始和分割线，注意前面多了两个 --，以 ------WebKitFormBoundary{随机字符串}-- 作为结束。

        进行文件传输时，在分割的请求体内容中，为传输的文件数据，以及其它参数。

        ------WebKitFormBoundary{随机字符串}
Content-Disposition: form-data; name="test"; filename="a.php"
Content-Type: application/octet-stream

<?php $_GET['a'];?>
------WebKitFormBoundary{随机字符串}
Content-Disposition: form-data; name="filename";

index.php
------WebKitFormBoundary{随机字符串}--

        Content-Disposition 后的 form-data; 默认存在，也可以省略。

        其它编码

        • ASCII 编码
        • URL 编码
        • GBK 编码
        • Unicode 编码
        • Base64 编码

        Base64 编码相比 URL 编码，编码后的信息压缩比更小，为 8/6，也就是用 8 bit 来表示 6 bit 的信息。Base64 编码通常采用 26 个大小写英文字母，数字 0-9、+、/、以及填充字符 =，共计 65 个字符。

        由于 Base64 编码中包含 +，/，在 HTTP 请求中可能造成歧义，所以大部分 Web 服务器，默认只对请求内容进行 URL 解码。

        Base64 的编码规则如下：

        • 每 6 bit 按照 64 进制转换为对应的字符
        • 如果最后不足 6 bit，存在两种情况
          • 剩余 2 bit，则填充 4 bit，用这 6 bit 转换为对应的字符，并在结尾填入两个填充字符 =
          • 剩余 4 bit，则填充 2 bit，用这 6 bit 转换为对应的字符，并在结尾填入一个填充字符 =
